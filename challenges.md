# allbets — Challenges Report

*Research synthesis, 2026-05-01*

The README scopes the current scaffold tightly (read-only public APIs, fuzzy matcher, three tools). The broader vision in discussion is wider: a wrapper that "enables agents to read **and write** to these individual markets." Most of what follows breaks the broader vision before it breaks the v0.1 scaffold — but a handful of things are already broken in the scaffold today.

## 1. Existential challenge: the cross-venue premise is weaker than the README assumes

The whole `pm_quote` / `pm_arb` value prop assumes the three venues list the **same questions**. They don't, in any meaningful sense, today.

I sampled the actual top markets on each venue:

- **Kalshi**, top-20 active by default order: 20/20 are `KXMVE…` Multi-leg Variant Exchange parlays — comma-separated lists of NBA / MLB player props plus game spreads (e.g. `"yes Reed Sheppard: 2+,yes Austin Reaves: 1+,yes Amen Thompson: 4+,no Houston wins by over 5.5 points"`). The field `market_type` is still `"binary"` even though the markets are 5-leg parlays.
- **Polymarket**, top-20 by 24h volume: BTC price, MegaETH FDV, Strait of Hormuz, LoL esports games, Fed June rate hike. All binary, 5/20 flagged `negRisk`.
- **Limitless**, top-20 active: 12/20 `Hourly`, 4/20 `15 min`, 4/20 `Daily`, all auto-generated by `automationType: "lumy"` against Pyth oracles. Examples: "Gold (PAXG) above $4558.99 on May 1, 11:00 UTC?", "BTC price on May 1, 11:00 UTC?". Zero political, zero sports, zero event markets in the top venues.

Of those 60 top markets across 3 venues, **the number that ask the same question on more than one venue is approximately zero**. The fuzzy matcher in `src/matcher.ts:51-80` (token Jaccard ≥ 0.45) won't find groups because there are no groups to find at the head of the volume distribution. You will get hits in the long tail (election / Fed / sports outcome events that all three list during major events), but the demo will be lifeless if you query the live API on a quiet day.

**Implication:** treat overlap-discovery as a first-class problem, not a post-fetch filter. A venue-aware *seed list* (e.g. crawl Polymarket's "trending events" → search Kalshi event listing → search Limitless slug-stable categories) will outperform "fetch top 10 from each, intersect."

## 2. The current Kalshi adapter is reading stale field names

`src/adapters/kalshi.ts:14-19` types `yes_bid`, `yes_ask`, `last_price`, `liquidity`, `volume`, `open_interest` (all integers in cents). The live `GET /trade-api/v2/markets` response actually returns `yes_bid_dollars`, `yes_ask_dollars`, `last_price_dollars`, `liquidity_dollars`, `volume_fp`, `notional_value_dollars`, `open_interest_fp` — Kalshi migrated to dollar-denominated and floating-point fields. **The adapter currently returns `0` for every price and `undefined` for every liquidity figure.** I confirmed this against the production endpoint. This is the single most urgent bug.

Also: the title field for the modal active market today is a 100-char comma-separated parlay leg list, not a question. Normalizing `title` straight to `question` (`kalshi.ts:33`) means the matcher will never group anything from Kalshi to anything else. You probably want to filter out `KXMVE*` tickers or special-case parlays.

## 3. Identity and credential models are completely incompatible across the three venues

For *write* operations (the broader vision), an agent needs three entirely different credential pipelines:

| Venue | Auth | Source of credentials | Per-request signing |
|---|---|---|---|
| Kalshi | Account (KYC'd) → web UI generates `.key` (RSA private key) + UUID API key ID | RSA private key file, downloaded once, **non-recoverable** ([docs](https://docs.kalshi.com/getting_started/quick_start_authenticated_requests)) | RSA-PSS-SHA256 over `timestamp+METHOD+path` per request, base64; 3 headers |
| Polymarket | Wallet → EIP-712 sign → derive L2 creds (apiKey/secret/passphrase) | Raw private key (EOA) **or** exported key from a Polymarket Magic-link proxy wallet **or** Gnosis Safe owner key | HMAC-SHA256 (L2) per request **plus** local EIP-712 sign per order; 5 headers ([docs](https://docs.polymarket.com/api-reference/authentication)) |
| Limitless | Scoped API token + EIP-712 | Token issued via UI; `verifyingContract` is venue-specific exchange address per market | HMAC-SHA256 per request; EIP-712 sign per order; 3 `lmts-*` headers ([docs](https://docs.limitless.exchange/api-reference/introduction)) |

There is no shared abstraction. An agent that holds "credentials for allbets" is really holding three secret bundles of different shapes. Agents that get keys from MCP env vars (the standard pattern) need three of them, and **Kalshi explicitly states its private key cannot be re-downloaded** — losing it costs the user a re-issue ceremony.

Worse for Polymarket: the `signatureType` (EOA / POLY_PROXY / GNOSIS_SAFE) and `funderAddress` differ depending on how the user originally onboarded. A user who signed up with email gets a Magic-link proxy wallet whose private key is buried in browser storage and must be exported manually before any programmatic trading is possible. **An MCP that promises "trade Polymarket" cannot do it for the largest user segment without a human onboarding step.**

## 4. Polymarket fragmented into two platforms in July 2025

Polymarket closed the **$112M acquisition of QCEX** (CFTC-licensed exchange + clearinghouse) on July 21, 2025 ([PRNewswire](https://www.prnewswire.com/news-releases/polymarket-acquires-cftc-licensed-exchange-and-clearinghouse-qcex-for-112-million-302509626.html)). Effective ~Q4 2025, Polymarket runs **two parallel platforms**:

1. The original Polygon-based CLOB — geo-blocked from 33 countries including the US ([geoblock docs](https://docs.polymarket.com/api-reference/geoblock)). Wallet-based, USDC-on-Polygon.
2. The QCEX-based CFTC-regulated US platform — KYC, USD funding, central clearinghouse.

The scaffold's adapter only knows the international Polygon Gamma API. For US-regulated agents (the Kalshi cohort), Polymarket means QCEX, which has a different API surface, identifiers, and likely a different market list. **A wrapper that just hits `gamma-api.polymarket.com` will miss everything tradable to a US user, and an arb tool that compares Kalshi US prices to Polygon-CLOB international prices is comparing prices in non-fungible markets** (you can't move USD from a US-KYC'd account to a Polygon wallet to capture the spread without weeks of lead time and several KYC ramps).

QCEX API documentation could not be verified publicly — it appears to be in private beta as of this report. This is a critical unknown for the road map.

## 5. UMA settlement risk is non-trivial for any agent holding positions

Polymarket settles via the UMA Optimistic Oracle. UMA disputes have flipped >$30M in market value during 2025 alone (Ukraine minerals, Fort Knox, UFO declassification — [polymarkets.co.il survival guide](https://polymarkets.co.il/en/guide/uma-disputes/)). One April 2026 controversy is referenced. The dispute window ranges from ~2 hours to weeks per market (`customLiveness` is per-market, exposed in Gamma response).

For the wrapper this means:
- A `pm_quote` "consensus" that includes a Polymarket market still in dispute window is conditional, not final. The schema in `src/schema.ts:37-43` has no field for resolution-state.
- An agent can be told "this market resolved YES" and then watch it flip to NO 24 h later. Without exposing `umaResolutionStatuses`, `umaBond`, `umaReward`, and `customLiveness` from Gamma in the normalized output, the agent has no way to assess settlement risk.
- Kalshi's centralized settlement returns in seconds–minutes; Limitless settles to a deterministic Pyth print at a specific timestamp; Polymarket can take days. A "spread" reported across these three is comparing apples to fish.

## 6. Multiple market identifiers per logical market

Polymarket alone has **three identifiers per market**:
- Gamma `id` (numeric, e.g. `540816`)
- `slug` (URL component, mutable)
- `conditionId` (on-chain, used in CTF / orderbook)
- Plus two `clobTokenIds` (one per outcome, ERC-1155 token IDs)

The CLOB API doesn't accept Gamma IDs and vice versa. The scaffold's `polymarket.ts:12` types `id: string` only — no `conditionId`, no `clobTokenIds`. The moment you add a write path you'll discover the order endpoint wants `tokenID`, not `id`. Same architecture holds on Limitless (`address`, `conditionId`, `slug`, two `positionIds`).

For cross-venue matching, "the same logical event" requires reconciling at the *event* level, not the market level — Polymarket multi-outcome events expose `clobTokenIds` per outcome, Kalshi groups markets under `event_ticker`, Limitless uses a `stableSlug` like `gold-hourly-price` for the recurring market series. Token-Jaccard on the *question text* of one outcome is the wrong primitive.

## 7. negRisk markets need a separate code path

5 of Polymarket's top-20 markets by 24h volume are flagged `negRisk: true` — these use a different settlement contract (Negative Risk Adapter), separate fee schedule, and conditional-token mechanics. The scaffold ignores the field. For reads you can probably get away with it; for trading you cannot. The `negRiskRequestID`, `negRiskOther`, `feeType`, `feeSchedule`, `makerBaseFee`, `takerBaseFee` fields are all in the Gamma response and all matter for cost models in `pm_arb`.

## 8. Limitless's volume claim of $789M needs a giant asterisk

The active markets are dominated by `lumy` auto-generated hourly/15-min Pyth-tracking binaries (15/20 in the sample). The market sampled at 10:02 UTC today had `volume: 0`, `openInterest: 0`, `liquidityFormatted: "10.000000"` (10 USDC of liquidity). The headline volume is plausibly real but largely composed of high-frequency auto-generated short-horizon contracts that re-roll every hour — the average market lifetime is short and wash-trading by the platform's own bots ("AI traders" Limitless markets publicly) is plausible.

For the MCP this matters because:
- Liquidity-weighted consensus (`matcher.ts:104`) will be dominated by Limitless's own market-making bots if you don't filter `automationType: 'lumy'`.
- The "same question" heuristic will catch ETH price markets across all three venues during macro events, but Limitless's strike is exact (e.g. "BTC above $77256.18") and Kalshi/Polymarket's are usually round-number ($150k by Jun 30) — the resolution criteria are different even when the headline question is "the same".

## 9. Rate-limit asymmetry caps your fan-out

- Kalshi: token-bucket per-tier (Basic = 200 read tokens/sec, ~20 RPS; default request cost 10 tokens; no `Retry-After` header) ([Kalshi docs](https://docs.kalshi.com/getting_started/rate_limits))
- Limitless: **2 concurrent requests, 300 ms minimum delay** between calls per token — extremely tight ([Limitless docs](https://docs.limitless.exchange/api-reference/introduction))
- Polymarket: largely undocumented for Gamma; CLOB has separate limits

The `fanOut` in `src/index.ts:23-37` fires all three in parallel with `Promise.allSettled` — that's fine for the scaffold. But anything that does pagination (search → resolve top events → fetch orderbooks) will hit Limitless's 300 ms floor before the others, making Limitless the always-slowest leg and the bottleneck for an "instant" agent quote.

## 10. The MCP protocol is a bad fit for write paths

MCP tool calls are RPC-shaped: parameters in, JSON out, finite timeout. Trading on these venues isn't:

- **Long-lived state.** Open orders are durable; partial fills happen over minutes; UMA disputes over days. The tool surface needs a separate `pm_orders_status` poll model, not a `pm_place_order` that returns "done".
- **On-chain confirmation.** Polymarket and Limitless require a Polygon/Base tx for collateral and settlement. The agent has to wait for confirmation; the MCP tool needs to either block (and risk timeout) or return a tx hash for the agent to poll separately.
- **Idempotency.** MCP has no native idempotency-key. If the connection drops between the tool and Polymarket and you got no ack, the agent doesn't know whether the order was placed. The agent will retry. It will double-fill. You need `client_order_id` plumbing exposed at the tool surface.
- **Secrets.** The standard MCP secret-distribution pattern is env vars at server-start. That works for one user. For a wrapper that multiple agents use across multiple users, you need a per-call credential model — which the MCP spec doesn't natively define and the stdio scaffold can't support without a redesign.

## 11. Existing competitors and what the gap actually is

EXA search surfaced 7+ existing prediction-market MCP servers:

- **`shaanmajid/prediction-mcp`** (TypeScript, 10⭐, last push 2026-04-13) — Kalshi + Polymarket, **per-venue tools** (`kalshi_list_markets`, `polymarket_list_markets`), no normalized schema, no cross-venue match. This is the closest competitor.
- `pab1it0/polymarket-mcp` (Python, 7⭐) — Gamma read-only.
- `BAiSEDagent/polymarket-mcp` (JS, 0⭐) — claims trade support, single-author, last touched Feb 2026.
- `kalashdotai/mcp` (Kalshi only)
- `yakub268/kalshi-mcp` (Python, claims trading)
- `polysolmcp/polysolmcp`, `undiabler/polymarket-mcp` — single-venue Polymarket.

**No existing MCP covers all three venues.** No existing MCP exposes a normalized cross-venue schema. No existing MCP does consensus/arb. So the differentiation is real — but only if the matcher/normalization layer actually works on real markets (see #1 and #2). A wrapper that's "Kalshi + Polymarket + a third venue I added" without genuine cross-venue value is just shaanmajid's tool plus 33% more API surface.

## 12. Bot/automation policy — yellow flags, not red

- **Kalshi** explicitly supports API trading and has token-bucket rate-limit tiers (Basic→Prime) priced for high-frequency use. CFTC regulation requires the **account holder be the trader of record** — a wrapper that lets multiple agents share one user's API key violates this. There is no "agent-on-behalf-of-user" category in CFTC rules.
- **Polymarket** has no public bar on bots; the L2 API is explicitly designed for it. Risk: if QCEX-platform US trading inherits CFTC bot-disclosure rules, an MCP-driven agent will need to flag itself.
- **Limitless** has no published policy. The team actively promotes "AI agent traders" on the platform. Lowest friction, but also least durable.
- **Cross-venue arb itself** is not market manipulation, but coordinating fills across venues to create the appearance of liquidity (a thing arb bots accidentally do) is. The `pm_arb` tool's output is fine; an agent that uses it to spam orders across all three to *create* a spread is not.

## 13. Funding ramps are an unsolved agent problem

- Kalshi: USD via ACH/wire from a US bank account in the user's name. **No agent can fund this autonomously.**
- Polymarket international: USDC on Polygon. Requires an exchange off-ramp + bridge.
- Polymarket QCEX: USD via ACH (assumed; no public docs).
- Limitless: USDC on Base. Requires bridge from L1 or another L2.

A "consensus quote" tool (read-only) sidesteps this. A "place an arb across all three" tool (write) requires pre-funded balances on all three plus the cross-platform float to actually rebalance after settlement. **For agents this is a much harder problem than the API plumbing** — and it's why every existing trading-MCP is single-venue.

## 14. Schema gaps in the current scaffold

Beyond the bugs already noted, the `NormalizedMarket` shape (`src/schema.ts:7-26`) is missing fields the agent will want:

- `resolution_status` (open / closed / disputed / settled / paying-out)
- `resolution_source` (URL / oracle / centralized rules text)
- `dispute_window_seconds` (Polymarket only — for UMA)
- `event_id` and `event_question` (so multi-outcome events can be grouped before per-outcome markets)
- `tradable_outcome_ids` (Polymarket `clobTokenIds`, Limitless `positionIds`, Kalshi `ticker`)
- `min_order_size`, `tick_size`, `maker_fee_bps`, `taker_fee_bps`
- `chain` / `collateral_token` (Polygon-USDC vs Base-USDC vs USD-on-Kalshi — the agent needs to know its capital is non-fungible)
- `restricted_jurisdictions` (which IPs/users are blocked)

Without these, the unified view is misleadingly clean.

## Priority list for the hackathon (next 12 h)

1. **Fix Kalshi field-name drift** (`src/adapters/kalshi.ts:14-19, 22-50`). Use `last_price_dollars`, `yes_bid_dollars`, `yes_ask_dollars`, `liquidity_dollars`, `volume_fp`, `open_interest_fp`. Otherwise every Kalshi number you display is zero.
2. **Filter Kalshi MVE parlays** — drop `ticker.startsWith('KXMVE')` from `searchMarkets` and `listActive`, or add a `is_parlay` field that the matcher down-weights.
3. **Add `event_ticker` / `conditionId` / `address` to the schema** as a stable cross-venue identity primitive, not just `venue_market_id`.
4. **Make the matcher event-aware**: Polymarket exposes `events` array per market; Kalshi has `event_ticker`; group on these before fuzzy-matching question text.
5. **Skip Limitless `lumy` markets** for `pm_quote` consensus (they distort liquidity-weighted average) — surface them in `pm_search` only.
6. **Add a `resolution_status` and `dispute_open_until` field** so agents don't treat in-dispute Polymarket markets as final.

## Strategic challenges (post-hackathon)

1. The QCEX bifurcation will force you to decide whether "Polymarket" in the wrapper is Polygon-international or QCEX-US. You probably need both as separate "venues" with overlap suppression, not one.
2. Trading is genuinely hard because of identity, custody, and funding fragmentation — not because the APIs are difficult. If you do scope trading, scope it to a single (user, venue) pair per MCP-server-process, not a multi-user fan-out.
3. Cross-venue arb is real on event days (elections, FOMC, major sports finals) and almost nothing on quiet days. The product is event-driven; market the demo against an upcoming catalyst.
4. The fuzzy matcher needs an LLM judge before this is useful (already on the roadmap as v0.2). Token Jaccard on questions like "Will the Fed increase interest rates by 50+ bps after the June 2026 meeting?" vs Kalshi's `KXFED-26JUN-T5` won't match.
